## ğŸ§© Module Architecture

- Every module follows the same structure layered into `src/modules/<tier>/<module_name>/` with
  templates, services, and tests packaged together.
- Marketplace-ready modules share the settings module contract: declarative configuration schemas,
  override hooks, and idempotent installers. Use `src/modules/free/settings/` as the canonical
  example when building your own modules.
- CLI helpers (`rapidkit add`, `rapidkit modules`, `rapidkit license`) manage module discovery,
  validation, and activation so you can focus on business logic.

## ğŸ¤ Contributing

1. Fork the repository and create a feature branch.
1. Run `make dev-setup` to install dependencies, format, lint, and run smoke tests.
1. Run `make community-docs` to regenerate the community README and ensure keyword guards stay
   clean.
1. Update or add tests inside `tests/user-tests/community/` or `tests/core/` as needed.
1. Submit a pull request targeting the community repository. RapidKit maintainers review, merge, and
   slot the changes into upcoming releases.

Community pull requests land in this distribution first. Internal teams lift and integrate approved
changesâ€”no separate workflow required from contributors.
